1. Basic Concepts¶

This page describes the core concepts and principles of Reflex individually. Other pages go into more detail about different concepts and how they are tied together.

1.1. Events and the Event Manager¶

This section refers to the Event and EventManager objects.

An event system is a system where different events are processed by different handlers. It is up to the event system to invoke handlers when an event happens. Handlers that listen for a certain event in the system, are bound to that event.

In Reflex, events are represented by the Event object, and the EventManager object invokes the handlers for those events.

The event manager can also be used to create event bindings. In fact, event bindings need to be created in order for the event manager to actually do anything when an event happens.

1.2. Event Handlers¶

Handlers are callable objects that are used to handle specific events when they happen. Applications may use plugins with handlers so that the plugins react to events that happen in the application.

1.3. Event Bindings¶

Bindings are objects which represent the link between a specific event handler and a specific event. The Binding object stores a reference to an event handler and relating to the conditions under which the handler is invoked.

The object stores the name of the event, or the type of the event, that the event handler is supposed to process. The binding also stores the rules associated with it.

1.4. Reactors¶

Reflex provides a base for Reactors, which are objects containing a selection of handlers. Reactors are also given a basic interface for creating and removing event bindings.

1.5. Rulesets¶

Rulesets are classes that manipulate the bahviour of the event system when event bindings are created and removed, and when events happen. The default Ruleset does not alter the behaviour of the event system much, and is used for events which do not have a specific Ruleset defined for them.

1.6. Event Rules¶

Event Rules are conditions associated with a specific event binding. These rules are stored as a list of items.

When an event is being processed by the event system, if the event binding is one associated with the event being processed, the rules are checked by the system before invoking the handler. If the items in the rules list of the event binding match corresponding items associated with the event being processed, then the handler is called.

The event rules are also referred to as event options.
